# ADR: Guest Session Token Design and Rate Limiting
Date: 2026-06-19
Status: accepted

## Context

WeddingLens guests authenticate by submitting an access code, OTP, or nothing (public events). We need to issue a session credential after successful authentication, protect against brute-force code guessing, and allow event owners to revoke all active guest sessions instantly.

The system runs on a single VM. There is no Redis or distributed session store — all state is in PostgreSQL or in-process.

## Decision

**Guest JWTs with a `type: guest` discriminator claim.**

A guest session is represented as a signed JWT with:
- `sub`: the `event_id` (string UUID) — all guest requests are scoped to exactly one event
- `type`: `"guest"` — distinguishes guest tokens from photographer/owner tokens
- `jti`: random UUID per token — for future revocation lists if needed
- `iat` / `exp`: standard issue-at and expiry claims; default TTL is `GUEST_SESSION_IDLE_TTL_SECONDS` (24 hours)

Signed with the same `SECRET_KEY` + `ALGORITHM` used for owner tokens, but the `type` discriminator prevents a guest token from being accepted as an owner token and vice versa.

**Session revocation via `guest_access_revoked_at` timestamp.**

Instead of a token blocklist, revocation is implemented by recording a `guest_access_revoked_at` timestamp on the event row (plus setting `guest_access_enabled = False`). The `get_validated_guest_event` dependency rejects any token whose `iat` is earlier than this timestamp. This is O(1) and requires no additional table.

**Idle session refresh (sliding window).**

`get_validated_guest_event` returns a freshly-issued token with a new `exp`. The caller is responsible for returning `X-Guest-Token: <refreshed_token>` in the response so the client can replace its stored token. This gives a sliding-window idle expiry without storing server-side state.

**In-process per-IP rate limiter (`GuestRateLimiter`).**

A module-level `GuestRateLimiter` singleton tracks `(event_id, ip)` failure counts. After `GUEST_LOCKOUT_ATTEMPTS` (default 3) failures, the key is locked for `GUEST_LOCKOUT_DURATION_SECONDS` (default 15 minutes). This is intentionally in-process: it is fast, requires no infrastructure, and is sufficient for the single-VM deployment model. Trade-off: lockout state is lost on process restart.

**OTP auto-generation.**

`otp_code` is a 6-character uppercase alphanumeric code generated by the service layer at event creation time (for `magic-link-otp` events) or on first update to that mode. It is never settable by the event owner via the API — this prevents accidental weak codes. It is included in `EventOut` (owner-only view) but excluded from `EventPublicOut` (guest-facing slug endpoint).

## Options Considered

| Option | Pros | Cons |
|--------|------|------|
| JWT with `type: guest` claim (chosen) | No DB lookup for token validation, stateless, fast | Revocation requires timestamp comparison on every request |
| Opaque session tokens stored in DB | Simple revocation (delete row), no JWT complexity | Every authenticated guest request hits the DB for session lookup; adds a sessions table |
| Redis-backed sessions | Fast, supports exact revocation, industry standard | Requires Redis infrastructure, not available on single-VM MVP |

| Rate limiter option | Pros | Cons |
|---------------------|------|------|
| In-process dict (chosen) | Zero infrastructure, zero latency | Lost on restart; not shared across multiple workers |
| PostgreSQL-backed attempt log | Persistent, works with multiple workers | DB round-trip on every failed attempt |
| Redis-backed | Persistent, fast, multi-worker | Requires Redis |

## Consequences

- Any code that needs to authenticate a guest request must use the `get_validated_guest_event` dependency (not `get_current_user`).
- The `sub` claim in a guest token equals the `event_id` as a string — all guest endpoints that call this dependency receive the validated event object directly and should not re-fetch by ID.
- Endpoints returning guest-authenticated responses should add `X-Guest-Token: <refreshed_token>` to implement the sliding window.
- The in-process rate limiter is reset on server restart. This is acceptable for MVP but should be replaced with a persistent store before horizontal scaling.
- `EventPublicOut` (returned by `GET /by-slug/{slug}`) must never expose `access_code`, `otp_code`, or `owner_id`. All future guest-facing event schemas must follow this pattern.
- `otp_code` is system-generated — never accept it from the client in `EventCreate` or `EventUpdate`.
